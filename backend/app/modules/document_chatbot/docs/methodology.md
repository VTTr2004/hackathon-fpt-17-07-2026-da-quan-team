# Methodology — Document Chatbot (RAG)

## Pipeline

```
Source (CSV row | PDF/DOCX/PPTX/XLSX text)
  -> ingestion: record card (1 row = 1 chunk) OR overlapping text chunks (keep page/slide/sheet marker)
  -> embeddings: NVIDIA nv-embedqa-e5-v5 (passage), stored per startup
  -> retrieval: HybridIndex = RRF( dense cosine , BM25 )   [startup-scoped]
  -> optional rerank: LLM listwise (GPT-OSS-120B)          [config: RAG_USE_RERANK]
  -> generation: GPT-OSS-120B constrained to retrieved SOURCEs, with [SOURCE n] citations
```

- **Provider:** NVIDIA NIM (OpenAI-compatible), chat `openai/gpt-oss-120b`, embeddings `nvidia/nv-embedqa-e5-v5` (1024-d).
- **Startup isolation:** the index key is the `startup_id`; retrieval never crosses startups.
- **Graceful degradation:** if `NVIDIA_API_KEY` is unset or the API fails, the index is BM25-only and
  generation falls back to extractive (most relevant chunk). The pipeline never hard-fails.

## Retrieval eval (how we chose the default)

Corpus: 100 record cards from `investments_VC.csv`. Query set: 80 synthetic questions over a
random sample of 40 companies — for each, a **name-based** lookup (mentions the company) and an
**attribute-paraphrase** question (no name; distinctive market/city/year/funding), generated by
GPT-OSS-120B. Gold = the source row. Metrics: Recall@1/3/5 and MRR@10. Reproduce with
`backend/eval/rag/rag_eval.py`.

| Method | split | R@1 | R@3 | R@5 | MRR |
|---|---|---|---|---|---|
| BM25 (lexical) | attr | 0.850 | 0.925 | 0.950 | 0.888 |
| Dense (e5) | attr | 0.800 | 0.900 | 0.950 | 0.860 |
| **Hybrid (RRF)** | attr | 0.850 | 0.900 | 0.925 | 0.887 |
| **Hybrid + LLM rerank** | attr | **0.900** | **0.950** | 0.950 | **0.928** |
| BM25 | all | 0.925 | 0.963 | 0.975 | 0.944 |
| Dense | all | 0.900 | 0.950 | 0.975 | 0.930 |
| Hybrid | all | 0.925 | 0.950 | 0.963 | 0.943 |
| Hybrid + rerank | all | 0.950 | 0.975 | 0.975 | 0.964 |

Name-based queries saturate at 1.000 for every method (the name is a strong shared signal), so the
**attribute split (n=40)** is the discriminating one.

### Findings

1. **Hybrid ≥ Dense**, and hybrid is more robust: it never underperforms the better single signal by
   more than noise, and protects against dense missing exact tokens (IDs, amounts, place names) or
   BM25 missing paraphrases. → **Hybrid (RRF) is the default retriever.**
2. **Dense slightly trails BM25 here** because the corpus and questions share literal tokens
   (market/city/year/funding numbers) — a lexical-friendly setup. On more semantic/synonym-heavy
   user questions, dense (and thus hybrid's dense arm) typically pulls further ahead. Treat these
   numbers as a lower bound on dense's value.
3. **Rerank helps top-rank precision** (attr R@1 0.850→0.900, MRR 0.887→0.928) but **does not raise
   Recall@5**. So when we feed top-5 to the generator, rerank adds little to final-answer grounding
   while costing one extra LLM round-trip. → **Rerank is OFF by default (`RAG_USE_RERANK=false`)**;
   turn it on when a tight top-1/top-3 matters (small context budget, single-citation answers).

### Caveats

- Synthetic questions, small n (40 per split) — differences under ~0.05 (≈2 questions) are noise.
- Name-based split is saturated and non-discriminating.
- No dedicated rerank NIM is available on `integrate.api.nvidia.com`; "rerank" here is an LLM
  listwise reorder, which is heavier than a cross-encoder. A hosted reranker would likely improve
  the cost/quality trade-off.
